## 1. 项目愿景与目标

本项目旨在为 iBizSim 企业竞争模拟竞赛打造一套**多决策域、三模式、双角色**的智能决策辅助系统。系统将覆盖竞赛全流程，通过数据分析、智能推荐和AI辅助，帮助**用户（学员）**快速制定高质量决策，同时支持高级用户高效设计和分发决策策略，最终形成一个可持续的策略生态系统。

系统设计的两个核心原则是：**对学员要快捷傻瓜，对想独立设计方案的用户要高效灵活易用。**

## 2. 系统架构

### 2.1. 技术栈选型

系统将采用**混合模式（Hybrid Model）**架构，兼顾低成本快速迭代和强大的可扩展性。

**前端 (Frontend)** 采用 `Vite + React + TypeScript`，承载所有核心计算逻辑（约束验证、公式求解）、UI交互和数据可视化。部署于 GitHub Pages 或类似静态托管平台，零服务器成本，计算响应快。

**后端 (Backend)** 采用 `Node.js / Python`（待定），仅承载必须由服务端处理的业务：AI大模型调用（封装 API Key，避免前端暴露）、数据持久化（用户账号、方案模板、付费状态）、权限与付费控制（方案密钥、下载次数、积分）。

### 2.2. 核心数据模型：三层架构

系统核心是解耦的**三层数据模型**，确保了配置与计算的分离。

| 层级 | 名称 | 说明 | 谁填写 |
|:---|:---|:---|:---|
| **1. 状态层** | `EnterpriseState` | 企业当前状态，包含从竞赛报表中获取的所有初始数据（上期库存、市场份额、财务状况等）。 | 用户手动抄入或通过"一键导入报表"功能自动填充。 |
| **2. 方案层** | `DecisionPlan` | 用户的决策意图和策略配置，描述**如何得出**决策值（哪些格手动填、哪些格系统求解、求解范围）。 | 用户手动配置，或从方案市场/密钥导入。 |
| **3. 决策层** | `IBizSimDecision` | 最终生成的、可直接提交给竞赛系统的完整决策数据。 | 由前端计算引擎根据 `State` + `Plan` 自动计算生成。 |

### 2.3. 数据结构定义

具体 TypeScript 接口定义见代码库中的 `types.ts` 文件，此处仅作结构示意：

- **`EnterpriseState.ts`**: 包含 `lastPeriodFinancials`, `marketShare`, `productInventory` 等从报表获取的数据结构。
- **`DecisionPlan.ts`**: 包含 `productionPlan`, `marketingPlan` 等，其叶子节点为 `CellBehavior` (`{mode: 'manual' | 'solve' | 'ai_solve', ...}`)。
- **`IBizSimDecision.ts`**: 包含 `production`, `marketing` 等，其叶子节点为最终的 `number` 类型决策值。

## 3. 角色与权限体系

系统采用简洁的**用户/管理员**二元角色划分。

### 3.1. 角色定义

| 角色 | 身份说明 | 核心权限 |
|:---|:---|:---|
| **用户 (User)** | 所有学员和方案设计者。分为"免费用户"和"高级用户"两个等级。 | 使用模拟器、浏览方案市场、创建/保存方案（受限）。高级用户可解锁发布方案、AI模式等。 |
| **管理员 (Admin)** | 系统运营者，拥有后台管理权限。 | 管理所有用户（含手动开通高级版）、审核/管理公开方案、配置系统参数。 |

### 3.2. 高级功能开通方式

在当前没有支付接口的情况下，高级功能的开通由**管理员在管理后台手动操作**。系统在数据结构上保留完整的付费计划字段（订阅类型、到期时间、积分余额等），以便未来接入支付系统时无缝切换。

## 4. 页面布局设计

### 4.1. 用户端布局

用户端采用经典的**左侧导航 + 顶部状态栏 + 主内容区**仪表盘布局。每个决策域内部通过**标签页（Tabs）**切换子功能，实现功能内聚。

**整体结构：**

```
┌─────────────────────────────────────────────────────────┐
│  顶部导航栏 (Header)                                      │
│  [当前竞赛: 2025全国赛]  [第3期]  [用户: student@email]  [退出] │
├──────────┬──────────────────────────────────────────────┤
│ 左侧导航  │  主内容区 (Main Content)                       │
│ (Sidebar) │                                              │
│           │  ┌─────────────────────────────────────────┐ │
│ 概览仪表盘 │  │  内部标签页 (Tabs)                        │ │
│           │  │  [模拟器]  [方案设计]  [我的方案]          │ │
│ ━━━━━━━━ │  └─────────────────────────────────────────┘ │
│ 生产决策 ◀│                                              │
│ 市场决策   │  ┌─────────────────────────────────────────┐ │
│ 财务决策   │  │                                         │ │
│ 研发决策   │  │     当前标签页对应的内容区域               │ │
│ 人力决策   │  │                                         │ │
│           │  │  例如: 点击"模拟器"                       │ │
│ ━━━━━━━━ │  │  → 显示生产决策模拟器                     │ │

│           │  │                                         │ │
│           │  │  例如: 点击"方案设计"                     │ │
│           │  │  → 显示 DecisionPlan 编辑器               │ │
│           │  │    (配置每个格子的求解行为和范围)           │ │
│           │  │                                         │ │

│           │  │  → 显示已保存/已购买的方案列表             │ │

│  底部状态栏 (Footer)                                      │

└─────────────────────────────────────────────────────────┘
```

**导航层级说明：**

| 导航层级 | 位置 | 内容 | 交互方式 |
|:---|:---|:---|:---|
| **一级导航** | 左侧 Sidebar | 决策域（生产/市场/财务/研发/人力）+ 全局功能（概览/方案市场） | 点击切换大模块 |
| **二级导航** | 主内容区顶部 Tabs | 每个决策域内部的子功能（模拟器/方案设计/我的方案） | 点击切换标签页 |

**关键设计原则：**

每个决策域（生产、市场、财务等）都是一个**自包含的功能单元**，内部统一包含三个标签页。这意味着"方案设计"和"我的方案"是从属于各自决策域的，而非全局功能。唯一的全局方案入口是左侧导航中的**"方案市场"**，它是一个跨所有决策域的公共市场，用户可以在这里浏览和获取所有类型的方案。

## 5. 商业模式

系统采用 **Freemium（免费基础 + 付费高级）** 模型。当前阶段无支付接口，由管理员手动开通高级功能；系统在数据结构上保留完整的付费计划字段，以便未来无缝接入支付系统。

| 功能层级 | 免费用户 | 高级用户（管理员开通） | 方案作者收益 |
|:---|:---|:---|:---|
| 手动模拟器 | 无限使用 | 无限使用 | - |
| 公式推荐（内置策略） | 每期3次免费推荐 | 无限推荐 | - |
| AI辅助模式 | 不可用 | 无限使用 | - |
| 公开方案下载 | 每月5个免费额度 | 无限下载 | 作者获得积分/分成 |
| 私有方案（密钥分享） | 密钥持有者免费使用 | 同左 | 作者自行决定是否收费 |
| 方案发布权限 | 最多发布2个方案 | 无限发布 | - |

**方案分发机制：** 用户创建 `DecisionPlan` 后存储到后端，系统生成**方案密钥**。方案可选择公开（进入方案市场供所有人浏览下载）或私有（仅通过密钥分享）。管理后台可配置每个用户使用方案密钥的免费次数，超出后需付费下载。

## 6. 数据存储方案

| 数据类型 | 存储位置 | 理由 |
|:---|:---|:---|
| 用户账号 + 付费状态 | 后端数据库 | 必须持久化，跨设备可用 |
| 方案模板 (`DecisionPlan`) | 后端数据库 | 需要分享、市场功能和密钥管理 |
| 方案密钥 + 使用次数统计 | 后端数据库 | 付费控制必须服务端 |
| 企业状态 (`EnterpriseState`) | 后端数据库（按竞赛期存储） | 跨设备使用 |
| 决策记录 (`IBizSimDecision`) | 本地 IndexedDB + 可选同步后端 | 默认本地快速，需要时同步 |
| 竞赛规则配置 (`GameRules`) | 后端数据库 | 不同竞赛场景的规则不同，需集中管理 |
| 操作日志 / 使用统计 | 后端数据库 | 运营分析和付费统计 |

## 7. 开发策略与实施路线图

### 7.1. 开发策略

系统采用**"完整结构 + 渐进式填充"**的策略进行开发。架构先行，优先搭建好完整的前后端项目骨架和三层数据结构；模块化实现，逐个决策域实现其完整闭环（UI → 计算 → 反馈）；参考但不依赖，第三方 Excel 公式仅作为设计参考，核心求解逻辑（如约束方程最优解）由我们自行研发。

### 7.2. 实施路线图 (Roadmap)

| 阶段 | 核心任务 | 交付成果 |
|:---|:---|:---|
| **Phase 1: 基础框架搭建** | 初始化前后端项目；定义三层数据结构 TS 类型；实现用户认证基础（登录/注册）；搭建用户端和管理端的页面骨架（Sidebar + Header + Tabs） | Git 仓库、`types.ts`、可导航的空壳页面 |
| **Phase 2: 生产决策闭环** | 迁移现有生产决策模拟器到新框架；开发生产决策求解器（约束方程最优解）；实现生产决策的 `DecisionPlan` 编辑器和"我的方案"页面 | 可独立运行的生产决策完整功能 |
| **Phase 3: 核心功能完善** | 逐个实现其他决策域（市场/财务/研发/人力）；对接 AI 辅助模式 API；实现报表数据导入功能 | 功能完整的多决策域系统 |
| **Phase 4: 运营与生态** | 开发方案市场（公开浏览/密钥分享）；实现付费计划与积分系统（预留支付接口）；建设管理后台完整功能 | 支持方案分享和付费的完整生态 |
