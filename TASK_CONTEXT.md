## 项目背景

**iBizSim 连续排产规则计算器** 是一个为企业竞争模拟大赛设计的生产排产工具。目标用户是参赛团队中负责生产的队员，需要在比赛中快速计算9期排产数据、分析多班次产能、对比排班成本，从而做出最优生产决策。

项目以纯前端静态应用形式部署在 GitHub Pages，无需后端服务。

## 核心业务逻辑摘要

### 排产计算核心（engine.ts）

`engine.ts` 是整个项目最重要的文件，包含：

1. **期数计算**：计算第1-9期的产能、成本、利润
2. **班次管理**：支持多班次（白班/晚班/夜班）配置
3. **雇佣约束**：每期雇佣人数受上一期人数约束，变化幅度有限制
4. **产能计算**：根据人数、班次、设备数量计算实际产能
5. **成本计算**：人工成本、设备成本、库存成本综合计算

### 第四期雇佣联动逻辑（待开发）

第四期（P4）的雇佣人数会直接约束第五期（P5）的可选范围：
- P4雇佣人数滑块范围：90-130人
- P5约束预览：根据P4人数实时计算P5的最小/最大可选人数

## 已知约束与规则

1. **路由约束**：wouter 已打 patch 以支持 GitHub Pages 的 hash 路由模式，不可随意升级版本
2. **样式约束**：使用 TailwindCSS v4，不使用 v3 的配置文件方式
3. **组件约束**：UI组件必须使用 shadcn/ui，不引入其他UI库
4. **状态管理**：使用 React 原生 useState/useContext，不引入 Redux/Zustand
5. **数据约束**：所有排产规则数据来源于 rule.xls，不可随意修改计算公式

- **代码风格**：TypeScript 严格模式，函数式组件，避免 class 组件

## AI指挥官工作流 (Standard Operating Procedure)

> **此为最高优先级指令，任何AI实例在接手此项目时必须严格遵守。**

为了确保任务的可持续性、可追溯性和跨账号/跨实例的执行一致性，本项目采用“三文件”工作流系统。AI在每一次启动或恢复任务时，**必须** 按照以下顺序读取和理解这三个核心档案：

### 1. **`PROGRESS.md` (进度快照)**

- **目的**: 快速了解项目的当前状态、最新进展和下一步操作。
- **读取时机**: **最优先读取**。
- **核心内容**:
    - **元数据**: 当前快照版本、时间、最新Commit。
    - **总体任务列表**: 查看已完成和未完成的大功能模块。
    - **当前执行阶段**: 精确找到上次任务的断点位置和恢复指令。
    - **历史快照索引**: 追溯历史版本和重大变更。

### 2. **`TASK_CONTEXT.md` (本文件：任务上下文档案)**

- **目的**: 深入理解项目的业务背景、技术架构、设计决策和约束规则。
- **读取时机**: 在读取 `PROGRESS.md` 之后，执行任何实质性操作之前。
- **核心内容**:
    - **项目背景与技术架构**: 建立对项目的宏观认知。
    - **核心业务逻辑**: 理解关键功能的实现原理，尤其是 `engine.ts`。
    - **已知约束与用户偏好**: 确保所有操作符合项目规范和用户期望。
    - **历史对话摘要**: 了解过去的决策过程和用户指令。

### 3. **`TASK_QUEUE.md` (功能需求任务队列)**

- **目的**: 管理用户的具体需求，跟踪每个任务的生命周期。
- **读取时机**: 在理解了项目状态和背景之后，准备开始新工作时。
- **核心内容**:
    - **任务队列**: 查看当前待办、进行中和已完成的任务列表。
    - **需求详细描述**: 理解每个任务的具体要求和验收标准。
    - **执行进度日志**: 查看历史任务的详细执行步骤。

### **工作流执行准则**

1.  **启动/恢复**: 严格按 `PROGRESS.md` -> `TASK_CONTEXT.md` -> `TASK_QUEUE.md` 的顺序读取文件。
2.  **接收新需求**: 当用户提出新需求时，**必须** 首先将新需求格式化并添加到 `TASK_QUEUE.md` 中，提交到仓库后，再开始执行。
3.  **更新进度**: 在完成一个阶段性任务或暂停工作时，**必须** 更新 `PROGRESS.md`，创建一个新的快照版本，并记录详细的断点信息和下一步操作。
4.  **记录决策**: 任何重要的技术选型、算法变更或用户指令驱动的修改，都应在 `PROGRESS.md` 的“关键决策记录”中留下痕迹。
5.  **提交代码**: 所有文件（包括代码和上述三个 `.md` 文件）的变更都必须通过 `git` 提交到远程仓库，并附带清晰的 commit message。

**此工作流是确保项目长期健康、高效协作的基石。任何绕过此流程的直接操作都是不允许的。**
